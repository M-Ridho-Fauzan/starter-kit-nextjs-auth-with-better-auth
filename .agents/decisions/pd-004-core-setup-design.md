# PD-004 — Distribution Model & Core Setup Design

> Status: **Approved**
> Relates to: Phase 1 item 2 (Better Auth core setup + adapter interface)

---

## PD-004 Resolution

**Decision: Option (b) — CLI installer (`create-next-auth-starter`) with a template repo.**

**Rationale:**
- Best UX — user answers 5-6 questions (database adapter, OAuth providers, features), gets a ready-to-run project
- Config flexibility at install time — template is customized per user's choices before they ever open the project
- Clean separation: CLI package (`create-next-auth-starter`) is a thin shell; the template is the real project
- Future-proof: template can evolve independently from the CLI
- Matches the "inspired by Laravel Jetstream & Breeze" vision — those use the same model (CLI installer + starter template)

**Rejected alternatives:**
- **(a) Repo template clone** — forces all users to delete features they don't need; no install-time customization
- **(c) npm package** — over-engineered; our value is the project structure, not a library import; users would still need a template

## How PD-004 Affects Phase 1 Item 2 (Core Setup)

### Singleton vs Factory

**Decision: Singleton pattern.**

Since the CLI generates the project with a static `auth.config.ts`, the better-auth instance is created once at module load time:

```typescript
// src/auth/server.ts
import config from "../../auth.config";
import { betterAuth } from "better-auth";

export const auth = betterAuth(mapConfig(config));
export type Session = typeof auth.$Infer.Session;
```

No factory, no dynamic config loading — the template knows exactly where `auth.config.ts` lives.

### Route Handler — Generated by CLI

The file `app/api/auth/[...all]/route.ts` is fully generated by the CLI, not written by the user:

```typescript
import { auth } from "@/auth/server";
import { toNextJsHandler } from "better-auth/next-js";

export const { GET, POST } = toNextJsHandler(auth.handler);
```

The user never touches this file unless they need custom middleware.

### Auth Client — Generated by CLI

The file `src/auth/auth-client.ts` is generated by the CLI based on which plugins the user selected:

```typescript
import { createAuthClient } from "better-auth/client";

export const authClient = createAuthClient({
  plugins: [
    // generated based on user's feature selections
  ],
});
```

### Config Mapping Layer

The auth config (`auth.config.ts`) defines features in our own format. A mapper module translates it to better-auth's native options:

- `src/auth/mapper.ts` — `mapConfig(config: AuthConfig) => BetterAuthOptions`
- Called once by `server.ts` at startup

This mapper is the bridge between our config-driven API and better-auth's native API. It's internal (not exported to user).

### Files for Phase 1 Item 2

| File | Purpose |
|---|---|
| `src/auth/mapper.ts` | Maps `AuthConfig` → `BetterAuthOptions` |
| `src/auth/server.ts` | Creates better-auth singleton, exports `auth` + `Session` type |
| `src/auth/auth-client.ts` | Client-side auth client, exports `authClient` + hooks |
| `app/api/auth/[...all]/route.ts` | Next.js route handler (generated) |

### API Contracts

```
auth.config.ts
  │ defineAuthConfig()
  ▼
AuthConfig (typed)
  │
  ▼
src/auth/mapper.ts :: mapConfig(AuthConfig) → BetterAuthOptions
  │
  ▼
src/auth/server.ts :: betterAuth(BetterAuthOptions)
  │                        │
  │                        ▼
  │              app/api/auth/[...all]/route.ts
  │
  ▼
src/auth/adapters/index.ts :: resolveAdapter(DatabaseConfig)
  │
  ├── prisma.adapter.ts
  ├── drizzle.adapter.ts
  └── kysely.adapter.ts
```

### TypeScript Strict Compliance

- `mapConfig()` returns `BetterAuthOptions` — typed via `z.infer` or better-auth's own types
- `server.ts` exports `typeof auth.$Infer.Session` — better-auth's canonical session type
- Adapter factories return `unknown` (as already designed) — cast happens inside `mapper.ts`
- No `any`, no `as unknown`
